The Problem with WAF Log Analysis

If you run Akamai Web Application Firewall at scale, you already know the pain. The SIEM API delivers a firehose of security events — blocked requests, rule triggers, attack signatures — and by the time your team sits down to ask "what were the top attack patterns yesterday?" or "which client IPs drove the most blocked traffic last week?", the raw data is sitting in multi-gigabyte compressed archives that take minutes to open, let alone analyse.

"We had the data. We just couldn't get to the insight fast enough to act on it."

Manual log review is slow, inconsistent across team members, and doesn't scale when you're collecting up to 8 GB of compressed events every hour across multiple WAF configurations. We needed a pipeline that was fully automated, reliable enough to run unattended in the cloud, and fast enough to generate a complete top-10 breakdown report in seconds — not minutes.

The Solution: A Three-Stage Pipeline

The answer turned out to be surprisingly lean: two Python scripts and one self-contained HTML file. No dedicated report server. No database. No proprietary dependencies beyond the Akamai SDK.

Each stage has a single, well-defined responsibility. The collector writes raw compressed archives. The aggregator distils those archives into compact frequency maps overnight. The report generator — a standalone HTML file that runs entirely in your browser — picks up those maps and renders a polished PDF report in under a second.

Stage 1 — Reliable Hourly Collection

collector.py runs every hour via cron (or any cloud scheduler). It handles full API pagination automatically: each run fetches up to one hour of WAF events for every configured SIEM configuration ID, enriches each event with decoded attack data and parsed request headers, and writes the output to a gzip-compressed JSONL archive.

Smart cursor resumption

After each successful run, the script saves the Akamai API pagination cursor and a per-configuration timestamp to a lightweight registry file (report_info.txt). The next hourly run checks whether each configuration's cursor is still fresh. If it is, collection resumes from exactly where it left off — no duplicate events, no gaps. If the cursor is stale (older than the configured window plus a 10% grace margin), it falls back to time-window mode automatically.

Resilience against connection resets

Streaming gigabytes of event data over HTTP is not always smooth. When the remote host closes the connection mid-stream, the collector retries up to six times with exponential back-off (1 s → 2 s → 4 s → 8 s → 16 s → 32 s), resuming from the exact event it was on. If all retries fail, any events already written are preserved on disk rather than discarded, and the last good cursor is saved for the next run.

✅
Production-hardened defaults

Six retries, exponential back-off, per-config staleness checking, and partial-file preservation combine to make the collector robust enough to run unattended for months without manual intervention.

Stage 2 — Overnight Aggregation at Scale

By the time your team arrives in the morning, the previous day's data is already aggregated and ready. At 00:05 UTC a short wrapper script:

1

Creates a dated archive folder

All hourly .jsonl.gz files are moved into SIEM events/YYYY-MM-DD/ — one folder per calendar day, clean and auditable.

2

Runs aggregator.py against that folder

The aggregator processes all archives in parallel using Python's multiprocessing, with optional orjson (3–5× faster JSON) and isal (2–4× faster gzip). A typical 8 GB archive is reduced to a <10 MB .agg.json file in 30–90 seconds.

3

Prunes folders past the retention period

Any dated folder older than the configured retention window (30 or 60 days) is deleted automatically — no manual cleanup required.

💡
From 8 GB to <10 MB overnight

The aggregator reduces a full day of compressed raw events into a tiny frequency-map file. The report generator reads this file — not the raw archives — so report load time drops from several minutes to under a second regardless of how much traffic your WAF sees.

Stage 3 — Customised PDF Reports in Under a Second

The report generator is a single self-contained HTML file. Open it in any modern browser — no installation, no login, no data leaving your machine.

Pick your categories, generate your report

Drop one or more .agg.json files into the interface. The app immediately displays which configurations are loaded and their event totals. Choose exactly which Top-10 breakdowns you want in the report:

Click Generate PDF Report. Long values like User-Agent strings and rule messages wrap across multiple lines rather than being truncated, and multi-line rows are kept together on the same page.

⚡
Raw .jsonl.gz files still work

For smaller datasets or ad-hoc analysis you can drop raw gzip archives directly into the report generator. The decompression and aggregation happen live in the browser — useful when you need a report for a specific hour without waiting for the overnight aggregation job.

Cloud-Ready Out of the Box

The pipeline was designed from day one to run unattended on cloud infrastructure. There is no GUI dependency, no database, and no web server — just Python and cron.

☁ AWS EC2 / Lambda ☁ Azure VM / Functions ☁ GCP Compute / Cloud Run ⎈ Kubernetes CronJob 🐳 Docker Container

What cloud deployment looks like

📅

Scheduler

Linux cron on any VM. Alternatively: AWS EventBridge, Azure Logic Apps, GCP Cloud Scheduler, or a Kubernetes CronJob — all work without code changes.

💾

Storage

Archives land in a local folder by default. Mount an NFS share, S3 bucket (via s3fs), or Azure Blob (via blobfuse) to the output directory for managed cloud storage with zero script changes.

🔑

Credentials

Akamai EdgeGrid credentials are read from a single .edgerc file. Store it in AWS Secrets Manager, Azure Key Vault, or GCP Secret Manager and mount/inject it at runtime.

📊

Reporting

The report generator HTML file can be hosted on any static file service — S3 + CloudFront, Azure Static Web Apps, or GitHub Pages — while keeping .agg.json files on a private store.

Getting Your First Report

Once the pipeline has been running for at least one full day, generating a customised report takes four steps:

1

Open report_generator.html in your browser

No web server needed. Double-click the file or drag it into a browser tab. Works offline.

2

Drop .agg.json files onto the page

Pick one file for a single day, or multiple files to merge data across days or configurations. The aggregated totals update instantly.

3

Select your report categories

Check the fields you care about today — rule messages, attacking IPs, User-Agents, paths, actions. Six categories are pre-selected by default based on the most common perimeter-defence use case.

4

Click Generate and download

A formatted PDF is built and downloaded in under a second. When multiple configurations are loaded, each gets its own PDF inside a single ZIP file.

What's Inside the Report

Every generated PDF follows a consistent structure so your team always knows where to look:

  • Cover page — configuration name, report date, time period covered, and total event counts (blocked vs. monitored).
  • Blocked traffic section — Top-10 tables for each selected field, ranked by event count with percentage share.
  • Monitored traffic section — same tables for events in monitor mode, useful for tuning false-positive rates.
  • Long-value rows — User-Agent strings, rule messages, and compound rule lists wrap across multiple lines with no truncation. Wrapped rows are never split across page boundaries.
  • Page footers — report timestamp and page N / total on every page.
